# In-app purchase

**HIG source:** https://developer.apple.com/design/human-interface-guidelines/in-app-purchase
**Last verified:** 2026-04-21
**Applies to:** iOS · iPadOS · macOS · tvOS · visionOS · watchOS

In-app purchase is how you sell **virtual goods** — premium features, digital content, subscriptions — through Apple's secure StoreKit flow. Not for physical goods or services (use Apple Pay for those). Four types: consumable, non-consumable, auto-renewable subscription, non-renewing subscription.

## When IAP vs Apple Pay

| Use | Content |
|---|---|
| **In-app purchase** | Virtual goods, premium app features, digital content, subscriptions |
| **Apple Pay** | Physical goods, tickets, reservations, services, donations |

Getting this wrong triggers App Store rejection.

## The four purchase types

- **Consumable** — depletes on use (game currency, hints, boosts). User can buy again.
- **Non-consumable** — permanent unlock (remove ads, pro feature, unlock level pack). One-time.
- **Auto-renewable subscription** — periodic billing until cancelled. Cloud service, premium content tier.
- **Non-renewing subscription** — fixed duration, no auto-renew (season pass, limited-time content).

## Rules for the store UI

- **Let people experience the app before paying.** Free use, free trials, previews. Users invest when they see value.
- **Integrated shopping.** Your store feels like your app — same typography, color, tone — not a separate Stripe-like checkout.
- **Simple product names and descriptions.** Direct language. Titles that don't wrap.
- **Display the total billing price** for every product. Required.
- **Hide the store when payments are blocked.** Parental restrictions, regional payment issues — show explanatory UI or hide entirely.
- **Use the default confirmation sheet.** System shows a sheet when the user taps Buy. Don't modify, don't replicate.

## Family Sharing

Non-consumable purchases and subscriptions can be shared across up to five family members.

- **Mention Family Sharing prominently** where users learn about your content. Use "Shareable" or "Family" in product names where apt.
- **Custom messaging for purchasers vs family members.** First time a family member views shared content: "Your family subscription includes…"
- **Onboarding existing subscribers.** When the purchaser enables Family Sharing, the system notifies the family. Support the flow — don't duplicate the notification.

## Refund help

Sometimes users need help. Sometimes refunds.

- **Custom purchase-help UI in-app.** FAQs, alternative resolutions, contact support, direct refund link.
- **Offer alternatives first.** "Didn't receive the item? Tap here for immediate fulfillment." Still show the refund button.
- **Clear refund title.** "Refund" or "Request a Refund" — the system flow clarifies it's Apple handling it.
- **Recent purchase context.** Show the product name, description, image, and original date so users identify the right purchase.
- **Don't characterize Apple's refund policy.** Don't speculate on approval. Link to Apple's support page.
- **Don't hide refund behind scroll or extra taps.** Must be easily reachable.

## Auto-renewable subscriptions

### Surfacing value

- **Onboarding calls out subscription benefits.** Show value during first launch, not after a dozen sessions.
- **Strong CTA + clear terms.** Monthly price, trial duration, auto-renewal language.
- **Offer tiers and durations.** Weekly / monthly / annual + different feature tiers. Let the user pick what fits.
- **Free content / free trial** before subscription prompt. Freemium, metered paywall, or true free trial.
- **Prompt at relevant moments.** User hits a free-content limit, uses a premium feature, saves something for later.
- **Don't prompt existing subscribers to subscribe again** — they'll think their subscription has lapsed when it hasn't.
- **Multi-app / web subscribers**: offer sign-in alongside purchase so users don't double-pay.

### Signup screen required content

Apple requires all the following visible in-app before checkout:

- **Subscription name, duration, and what the subscription provides.**
- **Billing amount in the user's local currency.**
- **Free-trial terms** if any — including duration and auto-charge at end of trial.
- **"Restore Purchases"** for existing subscribers.
- **Links to Terms of Service and Privacy Policy.**

### Offer codes

iOS / iPadOS support offer codes for new / existing / lapsed subscribers. Two types:

- **One-time use** — unique code generated by App Store Connect. Good for small / restricted distribution.
- **Custom** — reusable codes (`NEWYEAR`, `SPRINGSALE`). Good for mass campaigns.

Custom codes are **alphanumeric ASCII only** — no Chinese, no Arabic, no special characters.

Tell users **how to redeem**. Custom codes can be redeemed via a redemption URL or in-app. One-time codes via URL, App Store, or in-app.

In-app redemption uses `SKOverlay.AppClipConfiguration` / modern StoreKit's `offerCodeRedemption(isPresented:)`. The system provides the redemption UI — you just present it.

### Subscription management

Let users **upgrade, downgrade, cancel** without leaving the app.

- **Subscription summary.** Current plan, next renewal date, price.
- **System-provided management UI.** `manageSubscriptionsSheet(isPresented:)` in SwiftUI launches Apple's standard flow. Don't rebuild it.
- **Easy cancellation.** Deep-nested cancel is perceived as dark-pattern.
- **Win-back offers when the user cancels.** StoreKit notifies your app of cancellation — use the moment to offer a retention discount or ask why they're leaving.
- **Exit survey** (optional). Non-blocking. Useful for product feedback.

## Platform specifics

### iOS / iPadOS / macOS / tvOS / visionOS

Standard StoreKit flow. No additional considerations.

### watchOS

- **Same required info** as other platforms.
- **Modal sheet** to display the signup screen efficiently on a small screen. Default close button, scrollable content.
- **Two compact patterns for sub options:**
  1. **One button per option** — each button has locked-up description (duration, discount).
  2. **List of options + single signup button** — button title updates to reflect the selected option.
- **Clarify differences between devices.** If the watchOS app has different features than iPhone, say so.

## Common mistakes

- Selling physical goods via IAP (rejected).
- Selling virtual goods via external payment (rejected).
- Modifying or replicating the system confirmation sheet.
- Prompting existing subscribers to re-subscribe.
- Hiding refund / cancel flows.
- No "Restore Purchases" button.
- Missing billing total on the signup screen.
- Ambiguous free-trial terms (users surprised by first charge).
- Offer codes with Chinese or Arabic characters (invalid).
- Rebuilding subscription management UI from scratch (use StoreKit's).
- Up-selling in a Live Activity or notification (not allowed).
- Dark-pattern cancellation flows (users + App Store review don't tolerate).

## SwiftUI / StoreKit

```swift
import StoreKit
import SwiftUI

// Modern SwiftUI wrappers for the store and subscription UI.
struct PaywallView: View {
    @Environment(\.purchase) private var purchase

    var body: some View {
        SubscriptionStoreView(groupID: "com.example.pro") {
            VStack(spacing: 12) {
                Image(systemName: "star.circle.fill")
                    .font(.system(size: 64))
                    .foregroundStyle(.tint)
                Text("MyApp Pro").font(.largeTitle.bold())
                Text("Unlimited library, cloud sync, priority support.")
                    .multilineTextAlignment(.center)
                    .foregroundStyle(.secondary)
            }
        }
        .subscriptionStoreButtonLabel(.multiline)
        .subscriptionStorePolicyDestination(url: termsURL, for: .termsOfService)
        .subscriptionStorePolicyDestination(url: privacyURL, for: .privacyPolicy)
        .storeButton(.visible, for: .restorePurchases)
    }
}

// Present Apple's manage subscriptions sheet.
@State private var showingManage = false

Button("Manage Subscription") { showingManage = true }
    .manageSubscriptionsSheet(isPresented: $showingManage, subscriptionGroupID: "com.example.pro")

// Offer code redemption sheet.
@State private var showingRedeem = false

Button("Redeem Code") { showingRedeem = true }
    .offerCodeRedemption(isPresented: $showingRedeem) { result in
        // Handle the result
    }

// Consumable purchase.
func buyHints() async {
    guard let product = try? await Product.products(for: ["com.example.hints.10"]).first
    else { return }

    let result = try? await product.purchase()
    switch result {
    case .success(.verified(let transaction)):
        await unlockHints(count: 10)
        await transaction.finish()
    case .success(.unverified(_, let error)):
        showError(error)
    case .userCancelled, .pending, nil:
        break
    @unknown default:
        break
    }
}

// Listen for transactions (including subscription state changes).
Task {
    for await result in Transaction.updates {
        switch result {
        case .verified(let transaction):
            await apply(transaction)
            await transaction.finish()
        case .unverified:
            // Could be tampered; don't grant access
            break
        }
    }
}
```
